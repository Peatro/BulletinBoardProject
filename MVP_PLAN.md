# BulletinBoardProject: текущее состояние и план до MVP

## Текущее состояние

Проект находится на стадии backend-скелета с частично реализованной бизнес-логикой.

Что уже есть:

- Spring Boot backend на Java 21
- JPA + PostgreSQL
- Liquibase-миграции для базовых таблиц
- JWT resource server
- интеграция с Keycloak для регистрации и логина
- базовые модули `auth`, `user`, `advertisement`, `category`, `security`
- OpenAPI / Swagger
- `docker-compose` для локального запуска Postgres и Keycloak

Что уже реализовано по функциям:

- регистрация пользователя через Keycloak и создание локальной записи пользователя
- логин через Keycloak
- создание объявления
- получение объявления по `id`
- получение списка объявлений
- публикация объявления
- базовые admin-операции над пользователями

## Ключевые проблемы на текущем этапе

### 1. Проект не доведен до цельного MVP-сценария

Сейчас есть отдельные рабочие куски, но нет завершенного минимального пользовательского потока "зарегистрироваться -> создать объявление -> управлять своими объявлениями -> просматривать опубликованные объявления".

### 2. Модуль объявлений не завершен

- не реализовано обновление объявления
- часть controller-методов возвращает `null`
- маршруты объявлений заданы с ошибками
- нет нормальной обработки прав доступа и бизнес-ошибок

### 3. Инфраструктура частично заготовочная

- `Dockerfile` пока является заглушкой и не собирает приложение
- отсутствует `gradle-wrapper.jar`, поэтому `./gradlew` не работает
- конфигурация запуска и окружения не оформлена как воспроизводимая поставка

### 4. Тестовое покрытие практически отсутствует

- в `src/test` только пустой smoke-test
- нет unit/integration тестов на auth, security, advertisement, repository-слой

### 5. Доменные модули развиты неравномерно

- `category` есть в persistence-слое, но нет полноценного API
- `image` и `comment` пока фактически только сущности без завершенного use case
- часть функциональности выглядит как задел на будущее, а не как закрытая MVP-область

## Общая оценка

Текущее состояние можно оценить так:

- фундамент хороший
- архитектурное направление понятное
- MVP еще не готов
- ближайшая цель не расширять систему вширь, а довести до конца один короткий и надежный пользовательский сценарий

## Целевой MVP

Минимально жизнеспособная версия проекта должна поддерживать следующий сценарий:

1. Пользователь регистрируется и логинится.
2. Пользователь создает объявление.
3. Пользователь редактирует или удаляет свое объявление.
4. Пользователь публикует объявление.
5. Любой пользователь может просматривать опубликованные объявления и карточку объявления.
6. Есть справочник категорий.
7. Приложение можно локально поднять одной понятной инструкцией.

Все, что не помогает закрыть этот сценарий, должно быть либо упрощено, либо вынесено за пределы MVP.

## План развития до MVP

## Этап 1. Стабилизация платформы

Цель: привести проект в воспроизводимое и предсказуемое состояние.

Задачи:

- восстановить Gradle Wrapper и добавить отсутствующий `gradle-wrapper.jar`
- проверить совместимость `build.gradle.kts` с фактической средой запуска
- заменить заглушечный `Dockerfile` на реальную сборку Spring Boot приложения
- оформить понятную локальную схему запуска backend + postgres + keycloak
- вынести секреты и чувствительные параметры в переменные окружения
- проверить корректность `application.yml` и стартовых настроек security
- подготовить минимальный `README` с инструкцией запуска

Результат этапа:

- проект собирается предсказуемо
- приложение можно запустить локально без ручного гадания
- окружение подготовлено для дальнейшей разработки и тестирования

## Этап 2. Завершение ядра объявлений

Цель: довести основной доменный модуль до рабочего состояния.

Задачи:

- исправить REST-маршруты в controller слое
- реализовать `updateAdvertisement`
- реализовать корректное удаление объявления по `id`
- добавить проверку владельца на update/delete/publish
- отделить доступ к опубликованным объявлениям от доступа к черновикам
- нормализовать HTTP-коды и исключения: `400`, `403`, `404`
- добавить глобальный exception handler
- при необходимости разделить DTO для create/update/read
- добавить получение объявлений текущего пользователя
- добавить базовую фильтрацию списка объявлений
  - хотя бы по категории, статусу и автору

Результат этапа:

- модуль объявлений становится завершенным с точки зрения MVP
- появляется полный сценарий управления собственными объявлениями

## Этап 3. Категории как обязательный справочник

Цель: закрыть обязательную зависимость объявлений от категорий.

Задачи:

- добавить API для получения списка категорий
- при необходимости сделать древовидную структуру категорий
- подготовить seed-данные категорий через Liquibase
- проверить валидацию `categoryId` при создании и редактировании объявления

Результат этапа:

- frontend или клиентское приложение может стабильно работать со справочником категорий

## Этап 4. Минимальный пользовательский контур

Цель: закрыть базовый пользовательский профиль и self-service сценарии.

Задачи:

- добавить endpoint `GET /api/users/me`
- добавить endpoint обновления собственных пользовательских данных
- отделить self-service методы от admin-only операций
- привести user API к безопасному и понятному контракту
- не отдавать наружу JPA entity напрямую
- ввести отдельные DTO для user response/request

Результат этапа:

- появляется внятный пользовательский API без утечки внутренних сущностей

## Этап 5. Жесткий контроль MVP-границ

Цель: убрать все, что тормозит выпуск первой рабочей версии.

Решение по модулям:

- `comments` вынести за рамки MVP
- `images` либо временно исключить из MVP, либо сделать максимально упрощенный вариант

Рекомендуемый вариант:

- в первом MVP не делать полноценную загрузку файлов
- оставить либо пустой список изображений, либо упрощенную привязку готовых `objectKey`

Результат этапа:

- команда не тратит время на второстепенные модули
- фокус остается на основном пользовательском сценарии

## Этап 6. Тестирование критического пути

Минимальный обязательный набор:

- unit tests для `AdvertisementService`
- controller tests для auth endpoints
- controller tests для advertisement endpoints
- repository tests для основных запросов
- один интеграционный тест на запуск приложения с Postgres через Testcontainers

Статус:

- `AuthService` unit tests: сделано
- `AdvertisementService` unit tests: сделано
- auth controller tests: сделано
- advertisement controller tests: сделано, routes и ownership-сценарии доведены до green
- repository tests: не сделано

Что важно проверить в первую очередь:

- запрет редактирования чужого объявления

Результат этапа:

Задачи:

- проверить и донастроить Swagger/OpenAPI
- добавить healthcheck endpoint

- настроить CORS при необходимости

Результат этапа:

## Приоритеты выполнения

Если делать коротко и прагматично, порядок должен быть таким:

1. Стабилизация сборки и запуска
2. Починка и завершение advertisement API
3. Категории как справочник
4. Self-service user API
5. Тесты на критический путь
6. Подготовка к демонстрации
7. Все остальное после MVP

## Что не стоит делать до MVP

- полноценную систему комментариев
- сложную работу с изображениями и файловым хранилищем
- расширенную модерацию
- сложную ролевую матрицу
- богатый поиск
- сортировки и фильтры второго порядка
- избыточный рефакторинг ради красоты

## Критерий готовности MVP

MVP можно считать готовым, если:

- проект стабильно собирается
- проект стабильно запускается локально
- регистрация и логин работают
- пользователь может создать, отредактировать, удалить и опубликовать свое объявление
- опубликованные объявления можно просматривать публично
- категории доступны клиенту
- критический сценарий покрыт базовыми тестами
- запуск и проверка описаны в документации
